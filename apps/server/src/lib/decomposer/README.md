# `lib/decomposer/` — MusicologicalRules version sweep

**Status: experiment surface.** New rules versions get added here as the decomposer prompt evolves. Older versions are kept on disk for rollback and A/B comparison, not as cleanup debt.

## What's in here

The decomposer reads a reference track's metadata and produces a structured musicological brief (genre, harmonic character, arrangement_sections, etc.) that downstream pipeline stages consume. The brief is generated by an LLM call whose system prompt is one of these `rules-vN.ts` files.

| File | Status |
|---|---|
| [`rules-v1.ts`](./rules-v1.ts) – [`rules-v5.ts`](./rules-v5.ts) | Historical — reachable only via env/DB override |
| [`rules-v6.ts`](./rules-v6.ts) | Historical — but **still imported** by [`decomposer.ts:18`](./decomposer.ts). Not orphaned. |
| [`rules-v7.ts`](./rules-v7.ts) | Historical — reachable only via env/DB override |
| [`rules-v8.ts`](./rules-v8.ts) | **Current production default** ([`decomposer.ts:35`](./decomposer.ts)) |

## Default + selection

`getRulesText` ([decomposer.ts:35](./decomposer.ts)) returns v8 by default. The version is selected by:

1. `DECOMPOSER_RULES_VERSION` env var (operator override), or
2. The `rulesVersion` column on the relevant DB row (per-decomposition pin), else
3. v8.

Only v8 fires in default production traffic.

## Why old versions stay on disk

The decomposer is one of the most behavior-sensitive surfaces in the generation pipeline. Each `vN → v(N+1)` bump changes the brief shape and the music that gets generated downstream. Three reasons to keep old versions:

1. **Rollback.** A v9 that produces worse music can be flipped back via env var without a deploy.
2. **A/B comparison.** Decomposing the same reference track under v7 and v8 reveals what the prompt change actually shifted.
3. **Provenance.** Past `Decomposition` rows reference older versions; the file makes the prompt that produced them inspectable.

## Don't try to "clean up" v1–v7

They are not dead code. v6 is imported. v1–v5 and v7 are reachable on demand. Removing any of them changes the historical-reproducibility property of the pipeline.

If you actually want fewer files on disk, the right move is to (a) confirm with Daniel that the rollback / A/B / provenance properties are no longer load-bearing, and (b) archive the old versions to a `_archive/` subfolder or a separate git tag — not delete them.

## Adding a new version

1. Copy the current default (`rules-v8.ts` → `rules-v9.ts`).
2. Make changes to the new file. Leave older versions alone.
3. Update `decomposer.ts` to add v9 to the lookup map and (when ready) flip the default.
4. Add the top-of-file pointer comment matching the existing pattern.

The pattern at the top of each file is one line, matching this repo's existing comment style:

```ts
// Experiment surface — current default decomposer rules version. See ./README.md for the version sweep contract.
```

Or for historical versions:

```ts
// Experiment surface — historical decomposer rules version. See ./README.md for the version sweep contract; default is v8.
```

See [`ASSESSMENT.md`](../../../../../ASSESSMENT.md) §2 and §4 at the repo root for the audit that confirmed the v1–v8 sweep is intentional, not sprawl.
